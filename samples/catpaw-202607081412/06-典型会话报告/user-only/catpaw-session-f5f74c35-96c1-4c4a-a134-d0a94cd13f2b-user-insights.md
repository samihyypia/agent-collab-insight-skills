该会话显示用户希望 AI 能跨越“写规格”和“跑系统”之间的断层。

## 5. 痛点与摩擦

1. 技术概念漂移：AI 容易把 Skills、Tools、Claude Code CLI、Claude Agent SDK 混淆，用户在 T10、T20 多次纠正。
2. 文档假设未验证：T24 指出多个文件假设 SDK Read 可直接读取 OSS URL，但缺少最小可运行验证。
3. Skills 参数契约不一致：T35、T36 中 `--pageSize`、`--includeRelations` 报错，说明脚本接口与调用期待不一致。
4. 多服务调试链长：T27-T33 涉及前端、Java、FastAPI、Agent Runner、系统事件、权限接口、历史展示。
5. 文件链路存在体验和安全问题：T39-T40 暴露 OSS CORS 和文件展示/下载需求。

## 6. 用户成熟度

成熟度：专家型系统集成协作者。

依据：

- 能提前定义服务职责边界，避免系统耦合。
- 能识别 AI 规格中的技术假设，并要求 spike 或替代路径。
- 能使用真实环境、curl、错误日志推动联调。
- 能把运行失败转化为产品化需求，例如“工具失败记录到 SQLite”（T37）。

## 7. 可复用最佳实践

1. 架构规划阶段明确“谁负责什么”，例如 T9 对 Java/FastAPI 的职责划分。
2. 对 AI 生成的技术假设做最小可运行验证，例如 T24 对 OSS URL Read 能力的质疑。
3. 对 Skills 设计引用既有样例和接口协议，见 T15。
4. 联调阶段直接提供 curl 和真实错误，见 T27-T31。
5. 将失败追踪纳入数据模型，见 T37。

## 8. 反模式

1. 规格文档中出现未经验证的 SDK 能力假设，容易污染 tasks 和实现。
2. Skills 参数缺少单元测试或契约测试，导致运行时才发现参数不兼容。
3. 多服务同时启动后再集中排错，若缺少链路日志，定位成本会快速上升。

## 9. 建议

1. 为 ClaudeAgentSDK、Skills、Platform API 建立端到端 spike 清单，先验证文件读取、SSE、工具失败、权限失败、历史恢复。
2. 给每个 Skill 增加 CLI 参数契约测试，覆盖常用参数和错误参数。
3. 将 Agent Runner 事件、前端消息、SQLite 记录和工具调用 ID 串联，形成可追踪链路。
4. 对文件上传后的展示、下载、CORS、清理策略建立独立小规格，避免混在聊天主流程中。
